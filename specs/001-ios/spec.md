# 功能规范：Forest Focus Timer (iOS)

**功能分支**: `[001-ios]`
**创建时间**: 2026-03-10
**状态**: 草稿
**输入**: 用户描述：为 iOS 系统开发一款类似森林模式的番茄钟应用程序

## 用户场景与测试 *(必填)*

### 用户故事 1 - 开始专注植树 (优先级：P1)🎯 MVP

用户希望开始一个专注时段（默认 25 分钟，可选择 15/25/45/60 分钟），期间屏幕显示一棵正在生长的树。用户需要专注于任务，树会随着时间推移而成长。

**优先级原因**: 这是应用的核心价值主张——帮助用户专注并通过视觉反馈激励持续使用。没有这个功能，应用就失去了存在的意义。

**独立测试**: 用户可以通过主界面选择时长并点击"开始"按钮完全测试此功能，并验证倒计时是否正常运作，树木是否在 5 个阶段间正确生长。

**验收场景**:

1. **给定** 用户在主界面，**当** 用户选择时长并点击"开始植树"按钮时，**那么** 应用进入专注模式并启动倒计时
2. **给定** 专注时段正在进行，**当** 时间结束时，**那么** 树木完成最后一个生长阶段并播放成功提示
3. **给定** 专注时段正在进行，**当** 用户点击暂停按钮时，**那么** 计时器停止但保持当前进度
4. **给定** 专注时段已暂停，**当** 用户点击恢复按钮时，**那么** 计时器从暂停点继续运行

---

### 用户故事 2 - 中途放弃植树 (优先级：P2)

用户可能在专注过程中决定不再继续，需要能够主动放弃当前时段。放弃后，正在生长的树会被摧毁并从屏幕上消失。

**优先级原因**: 提供退出机制让用户感到控制感，避免强迫体验导致负面感受。这是用户友好的必要功能。

**独立测试**: 用户可以在任何时间通过"放弃"按钮完全测试此功能并验证树木是否被摧毁且数据被保存为未完成。

**验收场景**:

1. **给定** 专注时段正在进行，**当** 用户点击"放弃"按钮时，**那么** 树木显示被摧毁的动画效果
2. **给定** 用户放弃了专注时段，**当** 查看历史时，**那么** 该时段被标记为"未完成"并保存记录
3. **给定** 专注时段进行中，**当** 用户切换到后台时，**那么** 计时器继续运行但在适当时候通知用户

---

 ### 用户故事 3 - 查看个人森林 (优先级：P1)🎯 MVP

用户希望查看自己成功完成的专注会话历史记录，以看到自己的努力成果并获得成就感。

**优先级原因**: 可视化的成就系统增强动机，鼓励持续使用。这是核心激励循环的关键部分。

**独立测试**: 用户可以导航到森林视图，查看所有已完成的树木及其种植日期，完全验证数据持久化和展示功能。

**验收场景**:

1. **给定** 用户有一些成功的专注会话，**当** 用户查看森林视图时，**那么** 所有已完成的树木按时间顺序排列显示
2. **给定** 用户在森林视图中，**当** 用户点击某个过去的专注时段时，**那么** 显示该时段的详细信息（日期、时长等）

---

### 用户故事 4 - 查看统计数据 (优先级：P2)

用户希望了解自己的专注习惯和使用模式，包括总体进度和近期表现。

**优先级原因**: 数据反馈帮助用户了解自身模式，设定目标并跟踪进步。增加应用的长期价值。

**独立测试**: 用户可以访问统计面板，验证所有指标（总树木数、总时间、今日计数、连胜记录）是否正确计算和显示。

**验收场景**:

1. **给定** 用户有历史专注数据，**当** 用户打开统计面板时，**那么** 显示累计种植的树木总数
2. **给定** 用户有历史专注数据，**当** 用户打开统计面板时，**那么** 显示总专注时间（小时/分钟格式）
3. **给定** 今天是新的日历日，**当** 用户打开统计面板时，**那么** 今日计数重置为 0，随着新会话增加而更新
4. **给定** 用户昨天和今天都完成了专注，**当** 用户打开统计面板时，**那么** 显示连续天数正确的连胜记录

---

### 用户故事 5 - 接收专注提醒 (优先级：P2)

用户希望在专注时段即将结束时收到通知，以及在专注完成后获得庆祝反馈。

**优先级原因**: 确保用户不会错过专注完成时刻，强化积极反馈循环。提升用户体验和留存率。

**独立测试**: 用户可以在后台测试通知功能，验证专注结束时是否收到提示且应用能准确唤醒用户。

**验收场景**:

1. **给定** 专注时段进行中且设备锁定，**当** 时间结束到达时，**那么** 用户收到本地推送通知告知专注完成
2. **给定** 专注刚刚完成，**当** 应用显示完成界面时，**那么** 提供明确的庆祝视觉元素和数据保存确认

---

### 边界情况

- 当用户在专注时段开始时立即退出时会发生什么？树应显示最小生长阶段后被摧毁。
- 系统如何处理长时间中断（如设备重启）的情况？使用 Core Data 持久化，应用重启后根据保存的时间戳恢复会话状态。
- 当用户设备处于低电量模式时，后台计时应该如何调整？应尽量减少对电池的影响但仍保持基本准确性。
- 如果用户在不同设备上登录同一账户（未来扩展），当前会话如何处理？仅本地存储，不支持跨设备同步。
- 系统在专注期间收到电话或其他通知时如何处理？专注应继续运行，但可提供选项选择是否中断。
- 用户返回应用时发现专注已完成如何处理？根据系统时间差计算实际经过时间，同步更新 UI 到完成状态。

## Clarifications

### Session 2026-03-10

- Q: 数据持久化与存储策略 → A: 使用 Core Data 进行本地持久化，支持应用重启后恢复会话状态
- Q: 树木生长阶段的视觉表现形式 → A: 2D 矢量插画风格，静态图片切换
- Q: 专注时长的可配置性 → A: 支持预设时长选项（15/25/45/60 分钟），用户可在开始前选择
- Q: 后台计时器的准确性要求 → A: 使用系统时间差计算，返回前台时根据实际经过时间同步更新 UI 和状态
- Q: 数据保留与清理策略 → A: 永久本地存储，但提供用户手动删除单个树木或清空全部历史的功能

## 需求 *(必填)*

### 功能需求

- **FR-001**: 系统必须允许用户启动专注时段倒计时，支持预设时长选项（15/25/45/60 分钟）
- **FR-002**: 系统必须在专注期间显示树木生长的视觉表示，包含 5 个清晰的生长阶段，使用 2D 矢量插画风格的静态图片切换
- **FR-003**: 系统必须允许用户在专注期间暂停计时器并在稍后恢复
- **FR-004**: 系统必须允许用户随时主动放弃当前专注时段
- **FR-005**: 系统必须在用户放弃专注时显示树木被摧毁的视觉反馈
- **FR-006**: 系统必须在专注时段成功完成后将树木保存到用户的个人森林
- **FR-007**: 系统必须在专注时段被放弃时记录该会话为未完成状态
- **FR-008**: 系统必须在应用处于后台时继续精确计时，使用系统时间差计算方式确保准确性
- **FR-009**: 系统必须在专注完成时发送本地通知
- **FR-010**: 系统必须以网格布局展示用户个人森林中的所有已种植树木
- **FR-011**: 系统必须显示累积统计数据（总种植树木数和总专注时间）
- **FR-012**: 系统必须显示当日统计数据（今日专注计数）
- **FR-013**: 系统必须计算并显示每日连胜记录（连续完成专注的天数）
- **FR-014**: 系统必须允许用户手动删除单个树木记录或清空全部历史数据

### 关键实体

- **专注会话 (Focus Session)**: 代表一次专注尝试，包含开始时间、预设时长、实际经过时间、状态（完成/未完成）、树木生长阶段等信息。使用 Core Data 持久化存储
- **树木 (Tree)**: 代表一次成功的专注结果，包含种植日期、对应会话信息等属性。支持用户手动删除
- **统计数据 (Statistics)**: 汇总数据聚合，包括总数、今日计数、连胜等指标

## 非功能需求 *(必填)*

### 性能需求 (章程强制)

- **NFR-001**: 冷启动时间必须小于 2 秒（从启动到可交互状态，iOS 15+ 真机测试）
- **NFR-002**: 所有动画必须保持 60 FPS 恒定帧率，无明显卡顿
- **NFR-003**: 内存使用峰值不超过 150MB
- **NFR-004**: 后台任务执行时间不超过 30 秒
- **NFR-005**: 95% 的专注会话能够从后台切换后准确完成而不丢失进度（基于系统时间差计算）

### 无障碍访问需求 (章程强制)

- **NFR-006**: 完整支持 VoiceOver 屏幕阅读器，所有交互元素必须有明确的 accessibilityLabel
- **NFR-007**: 支持系统动态字体（最大字号 200%），布局不崩坏
- **NFR-008**: 符合 WCAG 2.1 AA 标准

### 安全与隐私需求

- **NFR-009**: 所有数据本地存储，不收集用户个人信息
- **NFR-010**: 不包含第三方分析 SDK、广告或追踪像素
- **NFR-011**: 不强制账户系统，用户无需登录即可使用核心功能

### 可维护性需求

- **NFR-012**: 单元测试覆盖率不低于 80%
- **NFR-013**: 每个公共 API 必须有合约测试验证接口稳定性
- **NFR-014**: 代码通过 Xcode static analysis 无警告

## 成功标准 *(必填)*

### 可衡量的结果

- **SC-001**: 用户可以从启动到完成第一个专注会话在 30 秒内体验到完整价值
- **SC-002**: 95% 的专注会话能够从应用到后台切换后准确完成而不丢失进度（设备后台锁定 25 分钟后返回）

- **SC-005**: 统计数据在所有场景下（新建会话、放弃会话、完成会话后）始终准确反映真实状态
- **SC-006**: 冷启动时间 < 2 秒（真机测试）
- **SC-007**: 所有 UI 动画保持 60 FPS，无明显卡顿
- **SC-008**: VoiceOver 用户可以独立完成所有核心功能
- **SC-009**: 动态字体放大至 200% 时，界面无布局崩坏或文本截断
